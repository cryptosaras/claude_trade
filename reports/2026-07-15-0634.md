## CRITICAL METHODOLOGY FINDING: the backtester is not reproducible
Ran `--strategy vwap_revert_short --days 14` **twice, minutes apart, no code
change**:
- Run 1: n=228, PF **0.82**, net −2919
- Run 2: n=244, PF **0.72**, net −4230

Same command, materially different result. Cause: the backtest window is
**not pinned** — `end = now()` advances between runs and it reads a
**live-mutating DB** (new 1m candles each minute; `tickers`/funding snapshot
refreshed every collector cycle; regime & funding series re-queried). With only
**6 position slots**, a tiny shift in the earliest entries cascades chaotically
through slot occupancy across the entire 14d replay (butterfly effect at the
slot bottleneck) — hence a 16-trade swing over a few minutes of drift.

**Implications:**
- A single backtest run **cannot rank** strategies or parameters. Any past
  decision that leaned on one backtest number (incl. recent retires) carries
  this caveat — re-check the important ones once the harness is pinned.
- Today's cap sweep (below) is therefore **inconclusive**, not a rejection.
- **Fix to make next session (do not hot-edit blind):** add a fixed `--end`
  timestamp so the window is pinned, and snapshot (or freeze) the candle/funding
  data for the run. Optionally make the 6-slot allocation deterministic. Then
  require *two identical runs* before trusting any backtest number. This is the
  highest-leverage code change on the board — it restores trust in the whole
  evaluation loop.
