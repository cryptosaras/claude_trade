## THE FIX: the backtest window is pinned (c2ca58e)

Last session's finding was that the same command minutes apart gave PF 0.82/n228
and 0.72/n244 — a broken evaluation loop. Root cause confirmed and fixed.

**What was non-deterministic**
- `end = now()` advanced between runs, so the window itself moved.
- Every input was read from a live-mutating DB.
- The engine has **no randomness** (checked). So the 16-trade swing was pure
  sensitivity to input drift: a tiny shift in the earliest entries cascades
  through the 6 position slots across the whole replay. That matters, because it
  means **pinning the inputs is sufficient** — no slot-allocation rework needed.

**Changes** (all in `app/backtest/main.py`)
- `--end <ISO8601 UTC>` pins the window; `start = end - days`. Default stays
  `now()` for backward compatibility (the `/api/backtest` endpoint is untouched).
- Every input read is bounded by `end`: candles (1m + 1h), regime, funding.
- The candle bound also fixes a subtle one: the terminal close-out prices
  still-open positions at `df["c"].iloc[-1]`, i.e. **the newest candle in the
  DB**, which drifted between runs even with a pinned window.
- **The crux:** the funding gap-filler read `SELECT symbol, funding_rate FROM
  tickers`, and the collector *overwrites* `tickers` every cycle. This was the
  one source that survives window-pinning — fix everything else and the twin
  test still fails. Replaced with an as-of-`end` read from the append-only
  funding table. Bonus: as-of-end is strictly *less* lookahead than as-of-now.

**Deliberately not done** (kept the change isolated so verification is clean):
the broader funding lookahead — periods before funding collection began still
borrow a later rate. Fixing that in the same commit would have made it
impossible to tell whether a number moved because pinning worked or because the
funding model changed. Documented as a follow-up.

## Honest caveat: "pinned" is not "frozen forever"
`ts <= end` is equivalent to a snapshot only because the tables are append-only
— and that is **not quite true**. `backfill_funding` (and `sync_symbol` for
candles) insert **past-dated** rows when the collector discovers a *new symbol*
(seen live: "funding history backfilled for 1 symbols", 13:34Z and 21:46Z Jul
15). So the same pinned window can change if the universe gains symbols between
runs. Consequences:
- Twin runs minutes apart: safe (a backfill landing between them is unlikely).
- Comparing a number to one from days ago: **re-run the twin**, don't trust the
  old figure.
A true fix would be a physical snapshot or a `collected_at` column; not worth it
yet, but it's the reason to re-pin rather than reuse old numbers.
