# 2026-07-06 17:13 — VPS 170% CPU investigation + infra fixes (ops session)

Human asked: why is the VPS at 170% CPU, do we have bugs (e.g. the Backtest tab
job "running" for 16h), can we optimize without losing anything. This session
is ops/infra only — no strategy changes. A concurrent research session was
active the whole time (whale_absorb + pdhl_sweep backtests).

## Diagnosis (measured, not guessed)

- **The 170–200% CPU was two concurrent backtests** in the api container:
  `whale_absorb --days 21` (docker exec) and `/tmp/bt.py` (pdhl_sweep, detached)
  — the concurrent session's legitimate work saturating both cores. Not a
  runaway bug. Engine 12.5% avg CPU, collector 0.3%, db idle: chronic load is fine.
- **Backtest row #3 (all·14d) stuck "running" 16h was a real bug.** POST
  /api/backtest ran the job as a daemon *thread inside the uvicorn process*.
  The 2026-07-05 21:36 UTC autopull rebuild killed the thread; nothing
  reconciled the row, so the UI showed "running" forever and the launching tab
  polls /api/backtest/3 every 3s indefinitely. Marked failed in DB by hand;
  the code fix prevents recurrence.
- **Bonus root cause:** a CPU-bound backtest thread inside the API process
  holds the GIL → the whole dashboard/API goes sluggish whenever a UI backtest
  runs. Likely why the box "felt" broken during long runs.
- **Engine ↔ DB moved ~57GB in 13h**: load_candles refetched 74 symbols ×
  3500 1m rows + 450 1h rows every 20s tick (~292k rows/tick, ~1.2MB/s) when
  only ~1 row/symbol/min is new. Wasteful tax on the shared 2-core box.

## Fixes shipped (commit 0b9ce93; multi-agent audit: 19 confirmed findings, 1 rejected)

- **api**: backtests now run as a `nice -n 10` subprocess (no GIL contention,
  dashboard stays responsive, OOM would kill the child not uvicorn); startup
  reconciles orphaned 'running' rows → 'failed'; 409 if a backtest is already
  running (one at a time on 2 cores); `days`/`step_minutes` bounded
  (step_minutes=0 previously looped run_backtest **forever** — a plausible
  alternate cause of "stuck running" jobs); run_backtest error-dicts now land
  as status=failed, not 'done' with dashes.
- **engine**: incremental candle cache (fetch only past the cached tip with
  5min/2h overlap; full reload every 30min heals drift). Safe because the
  collector only ever rewrites from its last stored bar forward — exactly the
  engine's cached tip. **Validated against the live DB before deploy:
  incremental == full-query truth for 1m/1h + stale-tail resync.** Expect
  db+engine traffic to drop ~99% (watch NET I/O in docker stats).
- **backtest**: `--strategy all` no longer force-activates retired/paused
  strategies (they stole position slots from the live portfolio → biased
  by_strategy numbers; 'all' now measures the actual portfolio).
- **collector**: 1h sync only for group symbols + BTC (1h is only consumed
  there; halves MEXC request budget — cycle time scales with collected count,
  up to 900 syms). Heartbeat now records `cycle_seconds` so candle staleness
  is observable. Promotion still backfills 180d of 1h in ~3 requests: no data loss.
- **ops/autopull**: convergence now tracked in `ops/.applied_sha` (written only
  after success) — previously `git reset` ran *before* the rebuild, so a failed
  rebuild (network/OOM) left containers silently on old app/ code forever with
  strategies/config already new. Also flock: rebuilds can outlast the 2-min cron.
- **ui**: strategy selector no longer silently resets to "all" when the
  strategies table refreshes (likely how the accidental all·14d run got
  launched); single poll handle (no stacked pollers); errors shown instead of
  eternal "starting…".

## Deliberately NOT changed (flagged as separate tasks for the human)

- **Zombie positions**: if a symbol is rotated out of universe.yaml groups
  while a position is open, core.py:105 `continue`s past SL/TP/time-stop —
  the position never closes and eats a slot forever. Needs a small orphan-close
  path; touching live position management deserves its own validation cycle.
- **SL/TP gap scan**: with bars_per_check=1, bars missed during a rebuild or
  CPU-starved tick are never scanned — a spike-and-revert through the SL
  during the gap doesn't fire. Fix: scan bars since last-checked ts.
- Watchlist (minor, from audit, some unverified due to session limits):
  can_open re-queries positions per candidate (~100 small queries/tick);
  regime_at is O(n²) in backtests; no docker log rotation (~50MB/day, rebuilds
  rotate de facto); sync_tickers does 3 single-row transactions per symbol.
